▶ Quick Start

This tool runs all 14 statistical tests defined in NIST SP 800-22 Rev 1a on a binary sequence. Each test produces a p-value. A p-value below α = 0.01 indicates evidence of non-randomness.

Three steps: ① Load a sequence (file, paste, or generate) → ② Adjust parameters if needed → ③ Click Run All Tests

Step 1 — Provide a sequence

Three input methods are available:

  • File drop / browse — drag any file onto the drop zone or click it to open a file picker. The format is detected automatically (see Supported File Formats below).
  • Paste — type or paste a string of 0s and 1s directly into the text area. Spaces, tabs, and newlines are ignored.
  • Generate — click 1 000 bits or 10 000 bits to fill the text area with a pseudo-random sequence for demonstration.

Minimum length: 100 bits. NIST recommends at least 1 000 bits. Tests 13 & 14 require 10 000+ bits in practice.

Step 2 — Parameters (optional)

Three numeric inputs control specific tests. Default values follow NIST recommendations and work well for most inputs — see the Parameters section below for details.

Step 3 — Navigate results

After clicking Run All Tests, the sidebar shows all 14 tests with colour-coded indicator dots. Click any test to slide its detailed view into the right panel — you will see the p-value, key statistics, and a chart. The summary strip at the top counts PASS / FAIL / N/A across all tests.

📂 Supported File Formats

Any file can be loaded. The pipeline detects the correct decoding mode in this order:

FormatDetectionHow it is decoded
Binary file Extension: .bin .exe .jpg .pdf .gz .zip and others Read as raw bytes via ArrayBuffer. Each byte expands to 8 bits, MSB first. Never interpreted as text.
Plain 0/1 text File content contains only the characters 0 and 1 (whitespace ignored) Used directly as a bit string. Spaces and line breaks are stripped.
Base64-encoded Content matches the Base64 character set [A-Za-z0-9+/=] after whitespace removal Decoded to raw bytes via atob() first, then each byte → 8 bits. Base64 characters are never analysed directly as bits.

Byte-to-bit convention (MSB first): byte 0xAB = 10101011₂ → bit string 10101011. This matches the NIST SP 800-22 convention used in all 14 tests.

⚙ Parameters

ParameterDefaultRangeDescription
Block M1288 – 10 000 Test 2 (Block Frequency). Size of each block in bits. The proportion of 1s per block should be ≈ 0.5 for a random sequence. NIST recommends M ≥ 20 and at least 100 complete blocks, so the sequence should contain ≥ 100 × M bits.
Template m92 – 10 Tests 6 & 7. Length of the bit pattern searched for. Test 6 (Non-Overlapping) uses the aperiodic pattern 0…01 and advances the window by m after each match. Test 7 (Overlapping) uses the all-ones pattern 1…1 with a sliding window of step 1.
LC block50050 – 5 000 Test 9 (Linear Complexity). Block length for Berlekamp–Massey analysis. Each block is tested independently. NIST recommends 500 ≤ M ≤ 5 000. Smaller values increase block count but reduce sensitivity; larger values detect longer linear structure.

🔍 Reading Results

Status badges

PASS (green) — p-value ≥ 0.01. No statistical evidence of non-randomness at the 1% significance level.
FAIL (red) — p-value < 0.01. The sequence exhibits a pattern this test is sensitive to.
N/A (grey) — The sequence is too short to run this test. The panel shows the exact minimum required.

Multiple p-values

Test 10 (Serial) reports two p-values. Tests 13 and 14 report one p-value per state (±1…±4 and ±1…±9 respectively). The overall PASS/FAIL badge for these tests uses the minimum p-value across all states — the most conservative approach.

How conclusive is a failure?

For any truly random sequence, approximately 1 in 100 tests will fail at α = 0.01 purely by chance. A single isolated failure is not conclusive. Consistent failures across multiple tests — especially Tests 1, 2, or 3 — or a p-value extremely close to 0 carry significantly more weight.

p-value very close to 1.0

An unusually high p-value (e.g. > 0.9999) can also signal a problem: the sequence may be too perfectly balanced — for example a forced alternating 10101010… pattern. For genuinely random sequences, p-values are themselves uniformly distributed over [0, 1].

Limitations of this tool

NIST SP 800-22 Section 4 recommends testing at least 55 sequences of 1 000 000 bits each for a formal generator evaluation. This tool analyses a single sequence at a time, which is sufficient for a quick check but not for a rigorous certification.

🎯 Purpose

Randomness is a foundational requirement in cryptography, secure communications, and scientific research. A sequence that appears random may contain detectable statistical patterns — patterns that can be exploited by an adversary or reveal flaws in a random number generator.

This tool makes the NIST SP 800-22 battery of tests accessible for:

  • Security researchers validating hardware TRNGs and software PRNGs before deployment.
  • Students and educators in cryptography or statistics — visual charts make abstract concepts tangible.
  • Developers performing acceptance testing on entropy sources.
  • Anyone curious about whether a given bit sequence looks statistically random.

📂 File Loading Pipeline

Any file can be loaded via drag-and-drop or the file picker. The pipeline determines the correct decoding mode automatically:

  • Binary files (.bin, .exe, .jpg, .pdf, .gz, …) — read as raw bytes; each byte expands to 8 bits, MSB first. Never interpreted as text.
  • Base64-encoded files — detected by character set; decoded to raw bytes via atob() first, then expanded to bits. Base64 characters are never analysed directly as bits.
  • Plain 0/1 text — used directly as a bit string. Whitespace and line breaks are stripped.
  • Unknown content — falls back to raw binary mode.

The byte-to-bit convention follows NIST: MSB first. Byte 0xAB (10101011₂) → 10101011.

⚠ Limitations

  • Tests 13 & 14 need ≥ 10 000 bits in practice to reach J ≥ 500 excursion cycles.
  • Test 8 (Maurer) needs ≥ 904 960 bits for the minimum L=6 parameter level.
  • NIST SP 800-22 Section 4 recommends testing ≥ 55 sequences of 1 000 000 bits each for a formal generator evaluation. This tool analyses one sequence at a time.
  • The Generate buttons use Math.random() — not a cryptographically secure RNG. Use for demonstration only.
