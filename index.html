Tests 1–14 · Significance level α = 0.01

▶ Quick Start

This tool runs all 14 statistical tests from NIST SP 800-22 Rev 1a on a binary sequence of your choice. Each test produces a p-value. A p-value below the significance threshold α = 0.01 is interpreted as evidence that the sequence is non-random.

Three steps:  ① Enter or generate a binary sequence  →  ② Adjust parameters if needed  →  ③ Click Run All Tests

Step 1 — Provide a binary sequence

Paste any string of 0s and 1s into the text area. Spaces, tabs, and newlines are ignored. The minimum accepted length is 100 bits; for reliable statistical conclusions NIST recommends at least 1 000 bits. Tests 13 and 14 (Random Excursions) need approximately 10 000+ bits to accumulate enough excursion cycles.

If you have no sequence at hand, use the Generate buttons to produce a pseudo-random sequence of 1 000 or 10 000 bits using the browser's built-in Math.random().

Step 2 — Set parameters (optional)

Three numerical inputs appear to the left of the Run button. They control specific tests only; all other tests ignore them completely. Default values follow NIST recommendations and work well for most inputs.

Step 3 — Interpret the results

Each test card shows a PASS (green), FAIL (red), or N/A (yellow) badge. The Summary bar counts all three categories. Cards include the exact p-value, key statistics, and a chart where applicable.

  • p-value ≥ 0.01 → PASS — no statistical evidence of non-randomness for this test.
  • p-value < 0.01 → FAIL — the sequence exhibits a pattern this test detects.
  • N/A → the sequence is too short; the card shows the minimum required length.

⚙ Configurable Parameters

ParameterDefaultRangeDescription
Block M 128 8 – 10 000 Block size for Test 2 (Block Frequency). The sequence is split into non-overlapping blocks of M bits. The fraction of 1s per block should be close to 0.5 for a random sequence. Larger M gives more statistical power but demands a longer input. NIST recommends M ≥ 20 and at least 100 blocks (so the sequence should contain at least 100 × M bits).
Template m 9 2 – 10 Template length for Tests 6 and 7. Test 6 (Non-Overlapping) searches for the aperiodic pattern 0…01 (m−1 zeros followed by one 1), advancing the window by m positions after each match so patterns never overlap. Test 7 (Overlapping) counts every occurrence of the all-ones pattern 1…1 with a sliding window of step 1. Larger m requires a proportionally longer sequence.
LC block 500 50 – 5 000 Block size M for Test 9 (Linear Complexity). Each block of M bits is analysed by the Berlekamp–Massey algorithm to find the shortest LFSR that can generate it. The deviation of the LFSR length from its expected value for random data is classified into seven categories and tested with a chi-square statistic. NIST recommends 500 ≤ M ≤ 5 000. Smaller blocks give more samples but each is less informative; larger blocks are more sensitive to linear structure.

🔎 Interpreting Results in Depth

What is a p-value?

Each test converts the observed sequence statistics into a single number — the p-value. It represents the probability that a perfectly random sequence would produce a test statistic at least as extreme as the one observed. A very small p-value means the observed pattern is highly unlikely to arise by chance.

Tests 13 & 14 — per-state p-values

Tests 13 and 14 compute a separate p-value for each state (±1…±4 for Test 13; ±1…±9 for Test 14). All individual p-values are listed inside the card. The card's overall PASS/FAIL badge uses the minimum p-value across all states (the most conservative choice).

When is a failure conclusive?

For any truly random sequence, approximately 1% of tests will fail by chance at α=0.01. A single isolated failure is not conclusive. Repeated failures across multiple tests — especially Tests 1–3 — or a p-value extremely close to 0 are more significant indicators of non-randomness.

p-value very close to 1.0

An unusually high p-value (e.g. > 0.9999) can also indicate a problem: the sequence may be too perfectly balanced — for example, a forced alternating 1010… pattern. Genuine random sequences produce p-values that are themselves uniformly distributed over [0, 1].

🎯 Purpose

Randomness is a foundational requirement in cryptography, secure communications, simulation, and scientific research. A sequence that appears random to casual inspection may contain subtle statistical patterns — patterns that can be exploited by an adversary or that reveal flaws in a random number generator (RNG).

The NIST SP 800-22 document defines a battery of statistical tests that probe different structural properties of binary sequences: frequency balance, run structure, linear complexity, pattern repetition, and random-walk behaviour. Together they form the de-facto standard acceptance test for RNGs used in cryptographic applications.

This tool was built to make that standard accessible to:

  • Security researchers and engineers who need to validate the output quality of hardware TRNGs, software PRNGs, or entropy sources before deployment.
  • Students and educators in cryptography, information theory, or statistics — the per-test charts and numerical breakdowns make abstract concepts tangible.
  • Developers implementing or auditing random number generators and needing a quick conformance check during development.
  • Anyone curious about whether a specific bit sequence — from a coin flip, a noise source, or an algorithm — looks statistically random.

🔨 Technical Notes

Mathematical core

All p-values are computed from either erfc(x) (complementary error function, Abramowitz & Stegun 7.1.26, accuracy < 1.5×10−7) or the regularised upper incomplete gamma function Q(a,x) (series expansion for x < a+1, Lentz continued fraction otherwise).

Important limitations

  • Tests 13 & 14 require at least ≈10 000 bits to yield J ≥ 500 excursion cycles — the minimum for a valid result.
  • Test 8 (Maurer Universal) requires ≥ 904 960 bits to engage the minimum L=6 parameter level. Shorter sequences return N/A.
  • This tool analyses a single sequence. NIST SP 800-22 Section 4 recommends testing at least 55 sequences of 1 000 000 bits each for a formal generator evaluation.
  • The Generate buttons use Math.random(), which is not a cryptographically secure RNG. Use generated sequences for demonstration only — not for security evaluation.
